## Why this spec exists

On 2026-07-12 (~02:30–08:00Z) the trigger scheduler degraded silently: **9
one-shot ticks were dropped** and **several cron failsafes wedged** with
`next_run_at` frozen hours in the past while still `enabled`. Two seats went
dark (Venture Lab, Self Improvement) because their failsafe was missing or
itself wedged. Everything needed to *catch* this was visible in
`list_triggers` all night — nothing was watching. The manager is the only
agent-side watchdog that can act (sibling sessions can't revive each other's
triggers — org policy), so the manager's wake ritual is where this check
belongs. Detection signature that would have caught it:
**`enabled = true ∧ next_run_at < now − 15min`.**

## The per-wake check (R26)

Each manager wake:

1. **Read the live trigger registry** (`list_triggers`, all pages).
2. **WEDGED-cron detection:** flag any trigger with `enabled = true` AND
   `next_run_at < now − 15min` as WEDGED. Record it in the roster (per-trigger
   health column) and in `control/status.md`.
3. **DROPPED-one-shot detection:** flag any `send_later` one-shot whose
   `run_once_at` is in the past and that is still `enabled` (never delivered).
   Group by the seat session it belongs to.
4. **DEAD-chain detection:** if a seat's pacemaker session has a dropped
   one-shot AND no future tick armed, treat that seat's chain as DEAD.
5. **Recovery:** for each DEAD-chain / dark seat, `send_message` that seat's
   session to resume and re-arm + verify its own chain. (`send_message` is the
   manager's only working cross-session revival path; fire/update/
   create_trigger on another session are org-refused. Manually firing a
   fresh-session trigger also works.) Where recovery is owner-only, raise it
   on the owner queue.
6. **Roster freshness is itself part of this:** if `docs/roster.md` is older
   than its regen threshold, the roster-regen cron may itself be wedged —
   flag it loudly (a stale roster hides the very silent-dark class this check
   exists to find).

**Done-when:** the check runs automatically each wake, and re-playing the
2026-07-12 registry snapshot (the venture-lab failsafe frozen at 06:06, the
kit-lab loop frozen at 06:08, the 9 dropped one-shots) surfaces every one of
them as WEDGED/DROPPED/DEAD within a single wake — the incident would have
been caught and a `send_message` recovery attempted the same cycle instead of
going unnoticed for ~6 hours.

### Notes (part of the spec)

- Distinguish, where possible, a QUEUED tick (bound to a busy session,
  delivers when the turn goes idle — sound by design) from a genuinely LOST
  one; if they're indistinguishable on the registry, flag both and let the
  owner/roster reader judge — an unlabeled queue is still a risk.
- Keep the health data on a second substrate (the roster is regenerated by a
  GitHub Actions cron, which survived the CCR outage) so the watchdog's own
  output isn't lost when the scheduler degrades.
- Do NOT re-edit `.claude/settings.json` to fix recovery-tool permission
  prompts — the exact CCR allow entries already exist and still prompt
  (superbot Q-0242, reproduced 2026-07-12). Any autonomous recovery duty must
  run in a Routine-spawned session whose spawn-time session_context carries
  the per-tool grants (the surface that provably works), not rely on
  settings.json.
